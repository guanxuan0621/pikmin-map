## Context

目前地圖頁面雖然已經支援定位與回到目前位置，但首次打開時仍停留在固定的預設區域。對不在該區域的使用者來說，這個初始狀態容易被誤解為地圖錯誤或資料不準。這次調整的重點不是改變瀏覽器權限模型，而是在地圖一進入時先用 app 內提示詢問使用者是否要定位，讓首次體驗更貼近實際所在位置。

## Goals / Non-Goals

**Goals:**

- 在首次進入地圖時顯示清楚的 app 內定位提示。
- 保持瀏覽器定位權限仍由使用者明確互動後才觸發。
- 讓使用者略過定位提示時仍能繼續正常瀏覽地圖。
- 避免提示在同一次地圖瀏覽中反覆跳出造成干擾。
- 若使用者先前已成功定位，重新整理後直接以最近一次成功位置開始。
- 使用者已定位時，小幅拖曳或縮放地圖應優先重用同一區域的附近蘑菇資料，而不是反覆觸發 API。

**Non-Goals:**

- 不在本次變更中改動地圖資料來源或預設底圖。
- 不在本次變更中引入新的瀏覽器權限 API 或背景定位。
- 不處理跨裝置記住提示偏好或永久記住不再顯示。
- 不在本次變更中實作持續追蹤位置更新或背景同步。

## Decisions

### Use an in-app first-open location prompt

地圖元件在初次進入且尚未取得目前位置時，顯示一個 app 內提示區塊，詢問使用者是否要以目前位置開始瀏覽。這個提示只負責引導，不直接等同於瀏覽器 geolocation 權限要求。

相較於頁面一載入就立刻呼叫 geolocation，這樣更符合目前規格裡「由使用者主動觸發定位」的原則，也能減少突兀的權限視窗。

### Keep geolocation permission user-triggered

真正的定位請求仍重用既有的 `handleCurrentLocationAction` 流程，只有在使用者按下提示中的定位動作後才發生。這樣既能保留現在的成功 / 失敗 / denied fallback 行為，也不需要拆出第二套 geolocation 邏輯。

### Limit the prompt to first map entry per page session

這個提示在同一次地圖頁面載入中只會出現一次。使用者若選擇稍後再說或已成功定位，就不再於同一輪瀏覽中重複顯示，避免在操作地圖時反覆遮擋。

### Reuse the last successful location on reload

當 geolocation 成功後，前端把最近一次成功的緯經度、精度與時間戳記寫入瀏覽器儲存。地圖元件初始化時若找到這筆資料，就先以它建立 `currentLocation` 狀態、同步目前位置標記，並把地圖中心設到該位置。這讓使用者重新整理後直接從上次成功位置開始，而且因為系統只是重用先前結果，不會再次觸發新的瀏覽器權限請求。

為了避免讓過久以前的位置誤導使用者，儲存資料會帶版本與時間戳記，讀取端只接受仍在有效期限內的紀錄；若資料缺失、過期或解析失敗，就退回既有的首次提示流程。

### Decouple nearby cache reuse from viewport jitter

目前快取 key 同時包含 viewport 邊界與定位中心，導致使用者明明還在同一個附近區域，只是輕微拖動地圖，就很容易產生新的 key 並重新打 `/api/mushrooms`。這次改成在「已有目前位置」的情況下，把快取視為一個以目前位置為中心、帶半徑的 nearby dataset；同一 nearby key 命中時，直接在前端依現在 viewport 過濾要顯示的蘑菇。

這表示 API 回來的 payload 不能只保存單次 viewport 已篩過的結果，而要保存對應 nearby 區域的完整資料集，再由 map client 依當前 bounds 決定畫面上的 markers 和列表。若沒有可用定位，則仍維持原本以 viewport 為單位的請求與快取行為。

## Risks / Trade-offs

- [提示過於顯眼造成地圖遮擋] → 使用非阻斷的 overlay 卡片，並保留略過選項。
- [提示與既有定位按鈕語意重疊] → 將提示定位動作直接串接既有定位流程，保持文案與回饋一致。
- [使用者略過後仍覺得預設區域奇怪] → 在提示文案中清楚說明可以稍後再定位，並保留既有定位按鈕。
- [保存的舊位置已經失準] → 為最近成功位置加上 TTL，過期後退回既有首次提示與預設地圖。
- [同 nearby 資料集不足以覆蓋使用者大幅拖曳的新視窗] → nearby cache 只在已定位且 key 命中的範圍內重用；一旦定位中心改變或 key 不同，仍會重新請求 API。

## Migration Plan

- 只需更新前端元件、前端快取格式與樣式，不涉及資料 migration。
- 若需回滾，可移除首次提示 UI，保留既有定位按鈕流程。
- 若需回滾最近成功位置功能，可移除對應 localStorage key，回到每次重整都顯示首次提示。
